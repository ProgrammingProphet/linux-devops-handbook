1. Linux Fundamentals basic

History · Architecture · Kernel · Distributions · FHS
Linux History From Unix to Linux
Timeline:

1969: Unix created at AT&T Bell Labs by Ken Thompson and Dennis Ritchie

1983: Richard Stallman launches GNU Project (GNU's Not Unix) - free Unix-like OS

1991: Linus Torvalds (Finnish student) creates Linux kernel v0.01

1992: Linux kernel released under GPL license

1993: Debian founded by Ian Murdock

1994: Red Hat Linux released

2004: Ubuntu released by Canonical

Today: Powers 90%+ of cloud infrastructure, Android, supercomputers

Why DevOps care?

Understanding Linux's open-source philosophy and Unix heritage helps grasp its design principles: everything is a file, small focused tools, text-based configuration.

What is Linux? OS components and architecture
Definition:

Linux is an open-source Unix-like operating system kernel combined with GNU tools and utilities.

Linux = Kernel + GNU Tools + System Libraries + Desktop Environment (optional)

Key Components:

Kernel: Core that manages hardware, memory, processes

Shell: Command-line interface (bash, zsh, sh)

GNU Tools: Core utilities (ls, cp, grep, etc.)

System Libraries: glibc, systemd

Package Manager: apt, yum, dnf, pacman

Why DevOps use it?

Free, stable, secure, highly customizable, dominates server/cloud infrastructure.

Linux Architecture System layers and components
Architecture Layers:

Layer 1 - Hardware: CPU, RAM, Disk, Network cards

Layer 2 - Kernel: Core OS, manages hardware resources

Layer 3 - System Calls: Interface between kernel and user space

Layer 4 - Shell & Utilities: Command-line interface and tools

Layer 5 - Applications: User programs and services

Kernel Responsibilities:

• Process management (scheduling, multitasking)

• Memory management (virtual memory, paging)

• File system management (ext4, xfs, btrfs)

• Device drivers (hardware communication)

• Network stack (TCP/IP, routing)

Example: 
uname -a

Show kernel version and system info

Linux Distributions Popular distros for DevOps
Major Distribution Families:

Debian-based: Ubuntu, Debian, Linux Mint

• Package Manager: apt, dpkg

• Use Case: Desktops, cloud servers, containers


Red Hat-based: RHEL, CentOS, Fedora, Rocky Linux, AlmaLinux

• Package Manager: yum, dnf, rpm

• Use Case: Enterprise servers, production environments


SUSE-based: openSUSE, SUSE Linux Enterprise

• Package Manager: zypper

• Use Case: Enterprise, SAP environments


Arch-based: Arch Linux, Manjaro

• Package Manager: pacman

• Use Case: Advanced users, rolling release

DevOps Popular Choices:

Ubuntu Server: Most popular for cloud/containers

RHEL/CentOS: Enterprise production servers

Amazon Linux: AWS-optimized

Alpine Linux: Minimal Docker images

Filesystem Hierarchy (FHS) Standard directory structure
Important Directories:

/bin - Essential user binaries (ls, cp, cat)

/sbin - System binaries (fsck, reboot, iptables)

/etc - Configuration files (nginx.conf, ssh_config)

/home - User home directories (/home/username)

/root - Root user's home directory

/var - Variable data (logs, databases, caches)

/var/log - System and application logs

/tmp - Temporary files (cleared on reboot)

/opt - Optional/third-party software

/usr - User programs and data

/usr/local - Locally installed software

/boot - Boot loader files (kernel, initrd)

/dev - Device files (sda, tty, null)

/proc - Virtual filesystem (process info)

/sys - Virtual filesystem (kernel/hardware info)

DevOps Important Paths:

/var/log - Check application logs

/etc/systemd/system - Service configurations

/opt - Deploy custom applications

/home/user/.ssh - SSH keys

Example: 
ls -la /

List root directory structure

Shell vs GUI Command-line interface
Shell (CLI):

Text-based interface for interacting with the OS

Popular Shells: bash (default), zsh, fish, sh

Advantages: Powerful, scriptable, remote access, automation, less resource-intensive

GUI (Graphical User Interface):

Desktop Environments: GNOME, KDE, XFCE

Use Case: Desktop users, visual applications

DevOps Note: Servers typically run without GUI to save resources

Why DevOps prefer CLI?

Automation, scripting, remote management, consistency, reproducibility

Example: 
echo $SHELL

Check current shell

cat /etc/shells

List available shells

2. Essential Commands basic

pwd, ls, cd, cp, mv, rm, cat, tail, grep, find, history …
pwd Print working directory
Description:

Shows current directory path.

Why DevOps use it?

Verify location before running commands, scripting navigation.

Syntax:
pwd [options]
Example: 
pwd

Output: /home/user/projects

DevOps use case:

In scripts: CURRENT_DIR=$(pwd)

ls List directory contents
Description:

List files and directories.

Why DevOps use it?

Daily navigation, checking logs, verifying files on servers.

Syntax:
ls [options] [file/dir]
Example: 
ls -la /var/log

-l = long format, -a = all files (including hidden), -h = human readable sizes

ls -lht

-t = sort by modification time (newest first)

DevOps use case:

Verify nginx log files: ls -lh /var/log/nginx/

Common mistake:

Forgetting -h makes sizes hard to read.

cd Change directory
Description:

Navigate between directories.

Why DevOps use it?

Essential for navigation, deployment scripts.

Syntax:
cd [directory]
Example: 
cd /var/log
cd ~

~ = home directory

cd -

- = previous directory

DevOps use case:

Quick navigation: cd /opt/app && ./deploy.sh

cp Copy files and directories
Description:

Copy files or directories.

Why DevOps use it?

Backup configs, duplicate files, deployment.

Syntax:
cp [options] source destination
Example: 
cp config.yml config.yml.bak
cp -r /source/dir /dest/dir

-r = recursive (for directories)

DevOps use case:

Backup before edit: cp /etc/nginx/nginx.conf /etc/nginx/nginx.conf.bak

Common mistake:

Forgetting -r when copying directories.

mv Move or rename files
Description:

Move or rename files and directories.

Why DevOps use it?

Organize files, rename configs, move logs.

Syntax:
mv [options] source destination
Example: 
mv oldname.txt newname.txt
mv /tmp/file.log /var/log/
DevOps use case:

Rotate logs: mv app.log app.log.old

rm Remove files or directories
Description:

Delete files and directories.

Why DevOps use it?

Clean up old files, remove temp data, disk management.

Syntax:
rm [options] file
Example: 
rm file.txt
rm -rf /tmp/old_data

-r = recursive, -f = force (no confirmation)

DevOps use case:

Clean old logs: find /var/log -name "*.log" -mtime +30 -exec rm {} \;

DANGER:

rm -rf / will destroy your system! Always double-check paths.

cat Concatenate and display files
Description:

Display file contents, concatenate files.

Why DevOps use it?

Quick file viewing, combining files, creating files.

Syntax:
cat [options] file
Example: 
cat /etc/hosts
cat file1.txt file2.txt > combined.txt
DevOps use case:

Check config: cat /etc/nginx/nginx.conf

tail Output last part of files
Description:

Display last lines of a file.

Why DevOps use it?

Live log monitoring, debugging production issues.

Syntax:
tail [options] file
Example: 
tail -f /var/log/syslog

-f = follow (live updates)

tail -n 50 app.log

-n = number of lines

DevOps use case:

Monitor errors: tail -f /var/log/nginx/error.log | grep 500

Common mistake:

Forgetting -f when you need continuous output.

head Output first part of files

📘 Description – Display first lines of a file.

🧠 Why DevOps use it – Preview files, check headers, quick inspection.

🛠 Syntax head [options] file
💻 Examples 
head -n 20 /var/log/syslog
🎯 DevOps use case – Check CSV headers: head -n 1 data.csv
grep Search text patterns
Description:

Search for patterns in files.

Why DevOps use it?

Filter logs, find errors, search configs.

Syntax:
grep [options] pattern file
Example: 
grep "error" /var/log/app.log
grep -r "TODO" /project/src/

-r = recursive, -i = case insensitive, -n = line numbers

DevOps use case:

Find errors: grep -i "exception" *.log

find Search for files and directories
Description:

Search for files by name, size, date, etc.

Why DevOps use it?

Locate files, cleanup old data, find large files.

Syntax:
find [path] [options]
Example: 
find /var/log -name "*.log"
find / -type f -size +100M

Find files larger than 100MB

DevOps use case:

Delete old logs: find /var/log -name "*.log" -mtime +30 -delete

history Show command history
Description:

Display previously executed commands.

Why DevOps use it?

Recall commands, audit actions, reuse complex commands.

Syntax:
history [options]
Example: 
history | grep docker
!123

Execute command number 123 from history

DevOps use case:

Find previous deployment command: history | grep deploy

3. File Permissions & Ownership intermediate

chmod, chown, umask, SUID, SGID, sticky bit
chmod Change file permissions
Description:

Change file permissions (read/write/execute).

Why DevOps use it?

Secure scripts, configuration files, and binaries on production servers.

Syntax:
chmod [options] mode file
Example: 
chmod 755 deploy.sh

Owner: rwx (7), Group: r-x (5), Others: r-x (5)

chmod u+x script.sh

Adds execute for user only (symbolic mode)

DevOps use case:

After deploying a binary: chmod +x /usr/local/bin/agent

Common mistake:

Using 777 on production files creates security risk.

chown Change file owner and group
Description:

Change file owner and group ownership.

Why DevOps use it?

Set correct ownership for application files, logs, and directories.

Syntax:
chown [options] owner:group file
Example: 
chown nginx:nginx /var/www/html
chown -R appuser:appuser /opt/myapp

-R = recursive (for directories)

DevOps use case:

Fix ownership after deployment: chown -R www-data:www-data /var/www/app

umask Set default file permissions
Description:

Set default permissions for newly created files and directories.

Why DevOps use it?

Control default security posture for new files.

Syntax:
umask [mask]
Example: 
umask 022

Files: 644 (rw-r--r--), Directories: 755 (rwxr-xr-x)

umask

Display current umask value

DevOps use case:

Set in shell profile for consistent permissions: umask 027

SUID / SGID / Sticky Bit Special permission bits
Description:

Special permission bits that modify file execution and directory behavior.

Why DevOps use it?

Control privilege escalation and shared directory access.

Types:

SUID (4): Run as file owner (e.g., passwd command)

SGID (2): Run as group owner, new files inherit group

Sticky Bit (1): Only owner can delete files (e.g., /tmp)

Example: 
chmod 4755 /usr/bin/myapp

Set SUID bit (4) + rwxr-xr-x (755)

chmod 1777 /shared

Set sticky bit (1) + rwxrwxrwx (777)

DevOps use case:

Shared upload directory: chmod 2775 /var/uploads && chgrp www-data /var/uploads

Security warning:

SUID on shell scripts is dangerous. Use with caution on binaries only.

4. User & Group Management intermediate

useradd, usermod, passwd, groupadd, su, sudo
useradd Create a new user account
Description:

Create a new user account.

Why DevOps use it?

Provision accounts for developers, CI/CD agents, or service users.

Syntax:
useradd [options] username
Example: 
sudo useradd -m -s /bin/bash -G docker jenkins

-m = create home dir, -s = shell, -G = additional groups

sudo useradd -r -s /bin/false appuser

-r = system account, no login shell

DevOps use case:

Create service account: sudo useradd -r -s /bin/false appuser

usermod Modify user account
Description:

Modify an existing user account.

Why DevOps use it?

Add users to groups, change shells, lock/unlock accounts.

Syntax:
usermod [options] username
Example: 
sudo usermod -aG docker username

-aG = append to group (don't remove from other groups)

sudo usermod -L username

-L = lock account

DevOps use case:

Add user to sudo group: sudo usermod -aG sudo username

sudo Execute commands as another user
Description:

Execute commands as another user (typically root).

Why DevOps use it?

Controlled privilege escalation, audit trail.

Syntax:
sudo [options] command
Example: 
sudo systemctl restart nginx
sudo -u postgres psql

-u = run as specific user

DevOps use case:

Grant user sudo access: edit /etc/sudoers or add to sudo group

Common mistake:

Editing sudoers directly instead of using visudo.

passwd Change user password
Description:

Change user password.

Why DevOps use it?

Set or reset passwords, enforce password policies.

Syntax:
passwd [username]
Example: 
sudo passwd username
passwd

Change your own password

DevOps use case:

Force password change on next login: sudo passwd -e username

groupadd Create a new group
Description:

Create a new group.

Why DevOps use it?

Organize users, manage shared resource access.

Syntax:
groupadd [options] groupname
Example: 
sudo groupadd developers
sudo usermod -aG developers john

Add user to the new group

DevOps use case:

Create deployment group: sudo groupadd deployers && sudo usermod -aG deployers jenkins

5. Process Management intermediate

ps, top, kill, systemctl, bg/fg, jobs
ps Report process snapshot
Description:

Report a snapshot of current processes.

Why DevOps use it?

Monitor running processes, find PIDs, troubleshoot resource usage.

Syntax:
ps [options]
Example: 
ps aux | grep nginx

a = all users, u = detailed, x = background processes

ps aux --sort=-%cpu | head -10

Find top 10 CPU-consuming processes

DevOps use case:

Find high CPU processes: ps aux --sort=-%cpu | head

top / htop Real-time process monitor
Description:

Real-time system resource monitor (htop is enhanced version).

Why DevOps use it?

Identify resource bottlenecks, high CPU/memory processes, system load.

Syntax:
top [options]
Example: 
top -o %MEM

Sort by memory usage

htop

Interactive, colorful, easier to use

DevOps use case:

Server slow? Check top for load average and CPU usage

kill Terminate processes
Description:

Terminate a process by PID.

Why DevOps use it?

Stop hung processes, restart services, manage runaway applications.

Syntax:
kill [signal] PID
Example: 
kill 1234

Graceful shutdown (SIGTERM)

kill -9 1234

Force kill (SIGKILL) - use as last resort

DevOps use case:

Kill all processes by name: pkill -f "python app.py"

Common mistake:

Using -9 first - it doesn't allow cleanup. Try graceful kill first.

systemctl Control systemd services
Description:

Control systemd services (modern init system).

Why DevOps use it?

Manage services, enable auto-start, check service status.

Syntax:
systemctl [command] [service]
Example: 
systemctl status nginx
systemctl restart docker
systemctl enable --now prometheus

Enable and start service immediately

DevOps use case:

App not responding: systemctl status myapp + journalctl -u myapp

bg / fg / jobs Manage background jobs
Description:

Manage foreground and background jobs in the shell.

Why DevOps use it?

Run long tasks in background, manage multiple processes in terminal.

Syntax:
command & (run in background)
Example: 
./long-script.sh &

Run in background

jobs

List background jobs

fg %1

Bring job 1 to foreground

DevOps use case:

Suspend with Ctrl+Z, then bg to continue in background

journalctl, dmesg, syslog, top, htop, vmstat, iostat

Query systemd journal logs.

Centralized logging for all systemd services, essential for debugging.

journalctl [options]

journalctl -xe

Debug failed service: journalctl -xe -u myapp

dmesg Print kernel ring buffer
Description:

Print kernel ring buffer messages.

Why DevOps use it?

Hardware issues, driver problems, kernel errors, boot diagnostics.

Syntax:
dmesg [options]
Example: 
dmesg -T | grep -i error

-T = human-readable timestamps

dmesg -w

-w = wait for new messages (follow mode)

DevOps use case:

Disk errors: dmesg | grep -i "I/O error"

vmstat Virtual memory statistics
Description:

Report virtual memory statistics.

Why DevOps use it?

Monitor system performance, CPU, memory, I/O, and processes.

Syntax:
vmstat [delay] [count]
Example: 
vmstat 1 5

Update every 1 second, 5 times

vmstat -s

-s = memory statistics summary

DevOps use case:

Monitor system load: vmstat 2 (continuous updates every 2 seconds)

iostat I/O statistics

free Display memory usage

Quick memory check, identify memory pressure.

uptime Show system uptime and load
Description:

Show how long system has been running and load average.

Quick system health check, verify uptime after reboot.

uptime [options]

sar System activity reporter

10. Cron & Task Scheduling intermediate

crontab, systemd timers, at
crontab Schedule recurring tasks
Description:

Schedule recurring tasks.

Why DevOps use it?

Automated backups, log rotation, health checks, cleanup jobs.

Syntax:
* * * * * command

(minute hour day month weekday)

Example: 
crontab -e

Edit crontab for current user

0 2 * * * /usr/local/bin/backup.sh

Run backup daily at 2 AM

*/5 * * * * /usr/local/bin/health-check.sh

Run every 5 minutes

0 3 * * 0 find /var/log -name "*.log" -mtime +30 -delete

Weekly log cleanup (Sunday at 3 AM)

DevOps use case:

Weekly log cleanup: 0 3 * * 0 find /var/log -name "*.log" -mtime +30 -delete

Common mistake:

Forgetting to redirect output: add >> /var/log/cron.log 2>&1

crontab -l / -r List or remove crontab
Description:

List or remove user's crontab.

Why DevOps use it?

View scheduled jobs, backup crontab, remove all jobs.

Syntax:
crontab [options]
Example: 
crontab -l

List current user's crontab

crontab -l > crontab-backup.txt

Backup crontab

sudo crontab -u username -l

View another user's crontab

DevOps use case:

Audit scheduled jobs: crontab -l

Common Linux DevOps interview questions with answers
Q: Difference between soft and hard links? File linking concepts
Answer:

Hard link: Points to inode directly, same file data, survives if original deleted.

Soft link (symlink): Points to filename, breaks if original deleted, can cross filesystems.

Example: 
ln file.txt hardlink.txt

Create hard link

ln -s file.txt symlink.txt

Create symbolic link

Q: How to check which process is using port 8080? Port troubleshooting
Answer: 
ss -tulnp | grep :8080

Modern approach using ss

lsof -i :8080

Alternative using lsof

Q: Difference between systemd and init? Init systems
Answer:

systemd: Modern, parallel startup, dependency management, socket activation, better logging (journald).

init (SysV): Traditional, sequential startup, simple shell scripts, slower boot.

Example: 
systemctl status nginx

systemd command

/etc/init.d/nginx status

Old init command

Q: What is SUID, SGID, and sticky bit? Special permissions
Answer:

SUID (4): Run as file owner (e.g., passwd command)

SGID (2): Run as group owner, new files inherit group

Sticky bit (1): Only owner can delete files (e.g., /tmp)

Example: 
chmod 4755 /usr/bin/myapp

Set SUID bit

chmod 1777 /tmp

Set sticky bit

Q: How to find files larger than 100MB? File search
Answer: 
find / -type f -size +100M 2>/dev/null

Find files larger than 100MB

find /var/log -type f -size +100M -exec ls -lh {} \;

Show details of large files

Q: Disk shows 100% but du shows space available. Why? Disk space mystery
Answer:

Deleted files still held open by processes. The space won't be freed until the process releases the file handle.

Investigation: 
lsof | grep deleted

Find deleted but open files

Solution:

Restart the services holding the deleted files or kill the processes

Q: Server is unreachable. What steps would you take? Troubleshooting scenario
Answer (Troubleshooting steps):

1. Ping server to check basic connectivity

2. Check cloud console/hypervisor for server status

3. Verify security groups/firewall rules

4. Check SSH service: systemctl status sshd

5. Review logs: journalctl -xe

6. Check network configuration: ip addr, ip route

7. Verify DNS resolution

Q: Application is slow. How to troubleshoot? Performance troubleshooting
Answer (Troubleshooting steps):

1. Check CPU usage: top or htop

2. Check memory: free -h

3. Check disk I/O: iostat -x 1

4. Check network: netstat -s

5. Application logs: journalctl -u app

6. Check database connections and slow queries

7. Profile application code

Q: Difference between TCP and UDP? Network protocols
Answer:

TCP (Transmission Control Protocol):

• Connection-oriented

• Reliable delivery with acknowledgments

• Ordered packet delivery

• Slower due to overhead

• Used for: HTTP, HTTPS, SSH, FTP


UDP (User Datagram Protocol):

• Connectionless

• No delivery guarantee

• No ordering

• Faster, lower overhead

• Used for: DNS, streaming, VoIP, gaming

Q: What is LVM? Logical Volume Manager
Answer:

Logical Volume Manager - flexible disk management system that allows:

• Resize volumes without downtime

• Create snapshots for backups

• Span volumes across multiple disks

• Move data between disks

Example: 
lvdisplay

Display logical volumes

lvextend -L +10G /dev/vg0/lv0

Extend logical volume by 10GB

Q: How to disable root SSH login? Security hardening
Answer:

1. Edit SSH config: /etc/ssh/sshd_config

2. Set: PermitRootLogin no

3. Restart SSH: systemctl restart sshd

Example: 
sed -i 's/#PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
systemctl restart sshd
Q: What is SELinux? Security framework
Answer:

Security-Enhanced Linux - mandatory access control (MAC) security mechanism.

Modes:

• Enforcing: Policies enforced, violations blocked

• Permissive: Policies not enforced, violations logged

• Disabled: SELinux turned off

Example: 
getenforce

Check current mode

setenforce 0

Set to permissive (temporary)
